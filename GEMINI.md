## 規劃階段角色說明

目前使用者的角色是計畫設計者，不是執行者。  
當前所有與學習計畫、Roadmap、專案結構、任務拆解、文件內容相關的對話，預設都應視為規劃討論，而非實際執行指令。  
除非使用者明確要求建立專案、產生檔案、撰寫程式碼或修改內容，否則僅提供規劃、建議、範本與文件草稿，不直接進入實作。

## Markdown 文件排版規範

當你建立、修改、重構本專案中的 Markdown 文件時，請遵守以下規範，目標是讓 VS Code 內建 Markdown Preview 能穩定正確渲染，並保持整份文件格式一致。

- 使用 `#`、`##`、`###` 作為標題，`#` 後面必須有一個半形空格，且標題前後保留一個空白行。
- 一般段落使用正常換行，不使用反斜線 `\` 強制換行；若同一段內需要換行，使用兩個半形空格結尾。
- 區塊標籤如 `**本週任務**`、`**實作清單**`、`**本週核心觀念**` 必須獨立成行，前後保留一個空白行。
- 無序清單一律使用 `-`，`-` 後面必須有一個半形空格；同一層級不可混用 `*` 或 `+`。
- Task list 一律使用標準 Markdown 語法：
  - 未完成：`- [ ]`
  - 已完成：`- [x]`
- Task list 的 `- [ ]` 或 `- [x]` 後面必須接一個半形空格再接文字，且中括號必須是半形字元。
- 子清單與子任務統一使用 2 個半形空格縮排，不使用 Tab，也不要超過 3 層巢狀。
- 清單前後都要保留空白行，不要讓清單緊貼在段落後面。
- 不要在段落與清單之間插入孤立的清單項目；同一區塊內的待辦事項必須集中在同一個 task list 中。
- 檔名、資料夾、指令、程式名稱一律使用行內程式碼格式，例如 `requirements.txt`、`src/`、`pytest`、`main.py`。
- 概念列舉使用一般無序清單，不使用 task list。
- 所有 Markdown 控制符號一律使用半形字元，包括 `#`、`-`、`[ ]`、`()`、`` ` ``，避免使用全形空格或全形中括號。
- 相同層級與相同類型內容的格式必須一致，不可混用不同風格。
- 在不改變原意的前提下，整份文件需統一格式，確保 VS Code 內建 Markdown Preview 能穩定正確渲染。
